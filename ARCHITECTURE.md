# ai-coach — Architecture

## 1. Overview

```
                  ┌─────────────────────────────────────┐
                  │           Cloud Server               │
    Slack DM      │  ┌──────────────────────────────┐   │
  ──────────────▶ │  │       Slack Bot              │   │
  ◀────────────── │  │  · event listener            │   │
                  │  │  · Scheduler                 │   │
                  │  └──────────────┬───────────────┘   │
                  │                 │                    │
                  │  ┌──────────────▼───────────────┐   │
    Terminal      │  │       Coach Agent             │   │
  ──────────────▶ │  │  · builds prompt             │   │
  ◀────────────── │  │  · calls Claude API          │   │
    (SSH / local) │  │  · refines skills            │   │
                  │  └───────┬──────────┬───────────┘   │
                  │          │          │                │
                  │  ┌───────▼──┐  ┌───▼─────────────┐  │
                  │  │ Memory   │  │  Skills Cache   │  │
                  │  │ Store    │  │  skills/*.md    │  │
                  │  └───────┬──┘  └───┬─────────────┘  │
                  └──────────│─────────│────────────────┘
                             │         │
                    progress sync    skills sync (daily)
                             │         │
                   user-configured   owner's skills repo
                     GitHub repo       (read + write)
                   (progress only)
```

The bot runs on a persistent cloud server. Both the Slack bot and the CLI connect to the same Coach Agent and share the same Memory Store and Skills Cache, so context and coaching quality are consistent regardless of interface.

Two daily sync jobs run independently:
- **Progress sync** — pushes `memory/progress.md` to the user's configured GitHub repository.
- **Skills sync** — pulls the latest skill files from the owner's skills repository. The owner's coach also pushes skill updates back; other instances only pull.

## 2. Components

### 2.1 Slack Bot

Listens for incoming Slack events (messages in the coach DM channel) and sends replies and proactive check-ins. Built with the Bolt SDK for Python.

### 2.2 CLI

A local terminal interface — a readline loop that sends input to the Coach Agent and prints the response. On startup it checks for any pending check-in prompts and surfaces them before the interactive session begins. Can be run directly on the server (via SSH) or packaged as a script the user runs locally.

### 2.3 Coach Agent

Receives a user message (or a scheduler trigger), assembles a prompt from the current session, `agent_state.md`, and relevant skill files, calls the Claude API, and returns the coaching response. Interface-agnostic — it does not know whether the caller is Slack, the CLI, or the scheduler.

When the user's feedback signals that an explanation worked well or poorly, the agent updates the relevant skill file in the Skills Cache. That change is pushed to the skills repository at the next daily sync.

### 2.4 Memory Store

Per-user files on the server. The agent reads `agent_state.md` on every turn; the session logs and `progress.md` are never loaded into the prompt.

| File | Written by | Read by | Purpose |
|---|---|---|---|
| `memory/agent_state.md` | Coach agent | Coach agent | Compact working memory: curriculum progress, observed learning style, open threads, digest of last session. Loaded into every prompt. |
| `memory/progress.md` | Coach agent | User | Human-readable progress summary. Synced to the user's GitHub repo daily. |
| `memory/sessions/YYYY-MM-DD.json` | Coach agent | Nobody (archive) | Full conversation log. Audit trail only — never loaded into a prompt. |
| `memory/config.json` | Ansible (deploy time) | Coach agent | Preferences: check-in frequency, progress sync target repo, skills repo URL, write-back enabled flag, model ID. Declarative — never written at runtime. |
| `memory/pending_checkins.json` | Coach agent (scheduler) | Coach agent (CLI/Slack on startup) | Queue of check-in messages generated by the scheduler and not yet shown to the user. Runtime state only — separate from config. |

### 2.5 Skills Cache

A local directory of Markdown files (`skills/*.md`) synced daily from the owner's skills repository. Each file captures the coach's accumulated knowledge of how to teach a specific topic — effective explanations, useful analogies, common misconceptions, sequencing notes.

Skills are topic-scoped (e.g. `mcp.md`, `software-factory.md`) and loaded selectively based on the current conversation. They are not user-specific and are shared across all coach instances.

### 2.6 Scheduler

Fires at the cadence stored in `config.json`, triggering the coach agent to generate a proactive check-in message and send it to the user via Slack.

### 2.7 Repo Sync

Two daily jobs:

- **Progress sync** — reads `memory/progress.md` and pushes it to the user-configured repository in `config.json`.
- **Skills sync** — pulls the latest skill files from the skills repository into the local Skills Cache. If `config.json` has write-back enabled (owner's instance only), also pushes any locally modified skill files back to the skills repository first.

## 3. Technology

| Concern | Technology |
|---|---|
| Messaging interface | Slack (Bolt SDK for Python) |
| CLI interface | Python readline loop (runs on server via SSH, or locally) |
| AI model | Claude API — configurable model ID (e.g. `claude-sonnet-4-5-latest`); set per instance in inventory |
| Bot hosting | Fly.io — Amsterdam region (AMS), EU data residency |
| Memory storage | Plain files (Markdown + JSON) on persistent Fly volume |
| Skills storage | Plain Markdown files; canonical copy in owner's GitHub repo |
| Proactive scheduling | APScheduler (in-process) |
| Repo sync | git push/pull via shell scripts |
| Runtime | Python 3 |
| Provisioning & deployment | Ansible (`ansible/`) |
| CI/CD | Thin GitHub Actions wrappers calling `ansible-playbook` |

## 4. Key Interaction Flows

### 4.1 User asks a question via Slack

1. User sends a message to the coach in Slack.
2. Slack delivers the event to the bot via webhook.
3. The coach agent loads `agent_state.md` and relevant skill files, then builds a prompt.
4. The agent calls the Claude API and receives a coaching response.
5. The bot sends the response back to the user in Slack.
6. The agent updates `agent_state.md` and the session log.

### 4.2 User asks a question via CLI

1. User starts the CLI (locally or via SSH to the server).
2. Any pending check-in prompts are displayed before the input prompt appears.
3. User types a message; the CLI passes it to the coach agent.
4. The agent loads `agent_state.md` and relevant skill files, calls the Claude API, and returns a response.
5. The CLI prints the response and awaits the next input.
6. The agent updates `agent_state.md` and the session log (same store as Slack).

### 4.3 Coach refines a skill

1. During a session the user signals an explanation worked well or poorly.
2. The coach agent updates the relevant skill file in the local Skills Cache.
3. At the next daily skills sync, the updated file is pushed to the skills repository (owner's instance only).
4. Other coach instances pull the update at their own next daily sync.

### 4.4 Proactive check-in

1. The scheduler fires at the configured cadence.
2. The coach agent reads `agent_state.md` and generates a check-in prompt.
3. The bot sends the prompt to the user as a Slack DM.
4. The user's reply is handled as a normal incoming message (flow 4.1).

### 4.5 Daily progress sync

1. A daily cron job reads `memory/progress.md` and the target repo from `memory/config.json`.
2. The job commits and pushes the file to the configured repository.
3. The file is now version-controlled and inspectable in that repo.

### 4.6 Daily skills sync

1. A daily cron job reads the skills repo URL and write-back flag from `memory/config.json`.
2. If write-back is enabled, locally modified skill files are pushed to the skills repository first.
3. The latest skill files are pulled from the skills repository into the local Skills Cache.

## 5. Deployment and CI/CD

### 5.1 Principle: Ansible for deployment, scripts for testing

Provisioning and deployment logic lives in an Ansible playbook (`ansible/site.yml`). GitHub Actions workflows are thin wrappers that do nothing except call `ansible-playbook`. This keeps the pipeline portable: any CI system, or a developer at their terminal, runs the exact same playbook.

Testing uses a shell script (`scripts/test.sh`) — Ansible is not the right tool for running pytest.

```
ansible/
  site.yml              # main playbook: provision → configure → deploy
  ansible.cfg
  roles/
    provision/          # create Fly app and volume if absent (idempotent)
    configure/          # render config.json, set Fly secrets
    deploy/             # flyctl deploy --app {{ fly_app }}
  inventory/
    owner.yml           # instance vars for the owner's deployment
    # colleague-*.yml   # one file per additional instance

scripts/
  test.sh               # install deps, run pytest

.github/workflows/
  ci.yml                # on PR        → calls scripts/test.sh
  _deploy.yml           # reusable     → ansible-playbook with given inventory
  deploy-owner.yml      # on push/main → calls _deploy.yml, inventory=owner.yml
```

### 5.2 Per-instance inventory

Each user's deployment is described entirely by an inventory file. The playbook and roles are generic and never edited per instance.

```yaml
# ansible/inventory/owner.yml
all:
  hosts:
    owner:
      fly_app: ai-coach-owner
      progress_repo: github.com/you/progress
      skills_repo: github.com/you/coaching-skills
      write_back: true
      checkin_frequency: daily
```

Adding a new instance = add one inventory file + add one caller workflow + add that person's secrets to GitHub.

### 5.3 Persistent storage

The `memory/` and `skills/` directories are mounted from a Fly.io persistent volume. They are not part of the Docker image and survive deploys. The `provision` role creates the volume on first run and skips it thereafter.

### 5.4 Secrets

Three secrets are required at runtime. Ansible's `configure` role sets them in Fly.io's secret store (never in the repository). In CI they are passed as GitHub Actions secrets, namespaced by instance:

| GitHub secret | Fly secret set by Ansible | Purpose |
|---|---|---|
| `OWNER_SLACK_BOT_TOKEN` | `SLACK_BOT_TOKEN` | Authenticates the bot with Slack |
| `OWNER_SLACK_APP_TOKEN` | `SLACK_APP_TOKEN` | Enables Socket Mode |
| `OWNER_ANTHROPIC_API_KEY` | `ANTHROPIC_API_KEY` | Authenticates calls to the Claude API |
| `FLY_API_TOKEN` | — | Allows GitHub Actions to call `flyctl` |
